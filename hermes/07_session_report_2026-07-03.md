# Отчёт о сессии выравнивания — 2026-07-03

> Что было сделано в проекте Skipper Logbook, в каком порядке и почему приняты
> именно такие решения. Ветка: `claude/skipper-logbook-ios-97z63d`, PR #1.

---

## 1. Задача

Привести существующий iOS-проект в соответствие с утверждённой продуктовой
концепцией **Liquid Nautical Minimalism**: аудит → Hermes-контекст → выравнивание
кода → верификация → документация. Ключевые принципы из ТЗ: никакой фейковой
функциональности, safety-first (MOB/якорь), умный дашборд состояний,
поддерживаемость для соло-разработчика.

---

## 2. Аудит репозитория

**Как:** три параллельных read-only агента (Core-слой; Features-экраны;
дизайн-система + виджеты + тесты + CI). Параллельно — чтобы за один проход
покрыть 3 200 строк фич и весь Core без потери деталей.

**Главные выводы:**

- Проект оказался **сильно более готовым**, чем предполагало ТЗ: реальные
  CoreLocation-движки (рекордер, якорная вахта, MOB), настоящие
  WidgetKit/ActivityKit-виджеты, палитра 1-в-1 с утверждённой, 21 юнит-тест.
- Но нашлись системные расхождения с принципами:
  1. **Фейковые данные в проде** — на первом запуске в реальное хранилище
     сидировался вымышленный флот («Sea Breeze», экипаж, 2 перехода, MOB-точка).
  2. **Подсистема waypoint/маршрута недостижима** — никто никогда не задавал
     координату назначения → линия маршрута, карточка «до точки», ETA были
     мёртвым кодом.
  3. **Якорная вахта не будила** — «тревога» лишь красила надпись.
  4. **MOB писался в журнал только на 1 из 3 путей запуска.**
  5. **Мёртвые контролы** — тумблер фоновой записи не мог сработать (никогда
     не запрашивался Always), пикер единиц ни на что не влиял.
  6. Аудиозаметки без тегов/скорости/курса и не в общей ленте журнала.
  7. Экрана «Погода» не было; экспорт — заглушка.

---

## 3. Hermes-папка

Папки не существовало — создана `hermes/` по предложенной в ТЗ структуре
(00-контекст, 01-направление, 02-дизайн-система, 03-план агентов,
04-план исполнения, 05-чеклист качества, 06-черновик release notes).

**Почему так:** соло-разработчику нужен один вход в контекст проекта; каждый
файл отвечает на один вопрос («что есть», «куда идём», «какие атомы UI»,
«кто за что отвечает», «что и в каком порядке делать», «как проверять»).

## 4. Alignment-пасс (коммит `93b9e37`)

| Решение | Почему так |
|---|---|
| Сид демо-данных убран из прод-запуска; оставлен за launch-аргументом `--seed-demo` (+превью/тесты) | Принцип «no fake data». Аргумент — потому что демо-набор полезен для разработки; превью и тесты используют собственные in-memory контейнеры и не пострадали |
| Добавлен CTA «Добавить судно» на пустом экране Vessel | Убрав сид, я убрал и единственный способ создать судно — сид был скрытой «регистрацией». Без CTA приложение стало бы неюзабельным |
| Waypoint ставится **тапом по карте** (режим по кнопке), а не формой ввода координат | На качке тап по карте — единственный реалистичный ввод; заодно оживает весь готовый код маршрута/ETA. Кнопка видна только при активном переходе — иначе была бы мёртвым контролом |
| MOB-события пишет **движок**, не вью (`MOBEngine.trigger` → журнал) | Единственный способ гарантировать «каждый путь логирует одинаково и ровно один раз». Дублирующая запись из TodayView удалена |
| Якорная тревога: хаптика + звук + локальное уведомление, «латч» с гистерезисом 80 % радиуса | Тревога, которая не будит — декорация (safety-first). Латч — чтобы дрожание GPS на границе радиуса не будило каждые 5 секунд |
| `.anchorDown`/`.anchorUp` пишутся в журнал | Полнота вахтенного журнала — базовое ожидание от судового журнала |
| Тумблер фона: включение запрашивает Always-доступ; подсказка, пока не выдан | Чинит мёртвый контрол по-настоящему, а не прячет его |
| Пикер единиц — `.comingSoon()` | Честность дешевле недоделки: прошивать unit-систему через все форматтеры без компилятора под рукой — высокий риск; вынесено в план следующей вехи |
| Теги аудиозаметок + скорость/курс; заметки вплетены в ленту журнала хронологически | Требование ТЗ; лента — потому что заметка в 10:04 должна стоять между событиями 09:58 и 10:12, а не жить в отдельной вкладке |
| Экран Weather: ручные наблюдения из журнала + явные Coming Soon для прогноза/приливов | ТЗ: «не фейковать живую погоду без источника». Наблюдения — реальные данные пользователя, всё остальное честно подписано |
| Экспорт CSV + GPX реализован по-настоящему; PDF остался Coming Soon | CSV/GPX — чистый Swift без зависимостей, входит в MVP-scope; PDF требует вёрстки и отдельной вехи |
| Today: баннер «MOB активен», карточка «На якоре», PermissionCard при запрете геолокации | Smart State Dashboard из ТЗ: экран обязан отражать состояние судна, а запрет геолокации — объяснять, а не показывать прочерки |
| Новые поля моделей (`tagsRaw`, `notes`, `waterCapacityLiters`…) — **только опциональные** | В проекте нет `VersionedSchema`, а `PersistenceController` падает `fatalError` при несовместимости. Опциональное добавление — единственная гарантированно лёгкая миграция SwiftData |
| Пустые `catch {}` в движках заменены на `os.Logger` | Молчаливо потерянный save во время MOB — худший вид отказа |

## 5. Adversarial-ревью собственного диффа

**Как:** 8 независимых агентов-«finder» (построчный скан; аудит удалённого
поведения; трассировка вызовов по файлам; reuse/конвенции; simplify/efficiency/
altitude), затем верификация. **Почему так:** в среде нет Swift-тулчейна —
компилятор заменялся перекрёстным ревью с разных точек зрения.

**Подтверждённые находки и решения (коммит `fb94e9f`):**

| Находка | Решение |
|---|---|
| MOB без GPS-фикса не записывался вообще (все 4 точки входа) | `trigger(from:)` в движке: без фикса пишет время инцидента в журнал и возвращает `false`; вью показывают алерт вместо пустого экрана поиска. Время — то, без чего поиск невозможен |
| Повторный MOB при активном плодил дубликаты и «терял» старую точку | Движок возвращает существующий инцидент: первая позиция и время — святое |
| Экспорт: снапшот на момент открытия экрана + коллизии имён файлов + генерация на main actor при каждом появлении | Генерация **в момент нажатия** (актуальные данные), в уникальную поддиректорию (UUID), через `UIActivityViewController`-мост — `ShareLink` требует URL заранее и вынуждал бы к преждевременной генерации |
| `allowsBackground` не переживал перезапуск | Персист в UserDefaults; восстановление в `init` не дёргает `didSet` → нет внезапного промпта на старте |
| Тревога мертва при заблокированном экране | `safetyBackgroundOverride`: при активной вахте/MOB фоновая геолокация включается принудительно (при Always-доступе) — безопасность выше настройки. Плюс честная подсказка на экране вахты |
| Записи resolve/тревоги (тип `.note`) не попадали в фильтр «Safety» | Новые типы событий `.mobResolved` / `.anchorAlarm` — полная лента инцидента там, где её ищут |
| Голосовая заметка писала **heading** (куда смотрит нос) в поле курса | Пишется course over ground — поле обещает курс |
| Дубли `logToLogbook`/`triggerMOB` в 4-х местах, мёртвые `AppState.anchorDown` и `isPDFAvailable` | Общий `LogEvent.record`, `recorder.setDestination`, `mob.trigger(from:)`; мёртвые символы удалены — «следующее поле добавляется в одном месте» |

**Findings пользователя** (все три подтвердились и вошли в тот же коммит):
P1 «поиск без точки» → возврат `Bool` + алерт; P2 «MOB не везде long-press» →
hold-to-activate на карточке Today и тайле Quick Actions с общей константой
`MOBButton.holdDuration`; P2 «локализация тела уведомления» → явный
`String(format: String(localized:))` с переименованным ключом — двусмысленность
исключена.

## 6. CI-сага: 8 прогонов, каждый вскрывал следующий слой

Репозиторий **ни разу не собирался в CI** до этой сессии — первый коммит был
сегодня же. Ошибки шли слоями, каждый фикс открывал следующий этап пайплайна:

| # | Симптом | Диагноз | Решение и почему |
|---|---|---|---|
| 1 | `future Xcode project file format (77)` | Brew поставил свежий XcodeGen, он генерирует формат Xcode 16.3+, а CI был на Xcode 15.4 | Пин `options.xcodeVersion` + нормализация `objectVersion` в CI. Попутно `SWIFT_VERSION: 5.9 → 5.0` — 5.9 не является валидным language mode и уронил бы сборку следующим |
| 2 | 7 × «main actor-isolated init in nonisolated context» | Swift 5.10 вычисляет default-значения `@State` вне актора | Синглтоны строятся внутри `@MainActor init()` App; `AudioRecorderController` получил `nonisolated init` (все дефолты полей безопасны) |
| 3 | `Cannot find type 'PersistentIdentifier'` на фазе emit-module | `AppRouter.swift` не импортировал SwiftData | +`import SwiftData`; свипом проверил весь проект на аналогичные пропуски (других нет) |
| 4 | **55** ошибок изоляции в хелперах вью | В SDK Xcode 15.4 у `View` изолирован только `body`; весь код (включая исходный) написан в модели Xcode 16+, где `View` целиком `@MainActor` | **Переезд CI на macos-15/Xcode 16** вместо обмазывания ~30 вью аннотациями: убирает класс ошибок на уровне SDK, соответствует реальности 2026 года. `DashboardReadout.make` (не-View) помечен `@MainActor` вручную. README/SETUP: минимум Xcode 16+ |
| 5 | `Text(date.shortDate)` + `ambiguous use of 'cos'` | `shortDate` — метод, 7 мест звали его без скобок (в интерполяциях молча рендерилось «(Function)» — унаследованный видимый баг!); тригонометрия Double×CGFloat стала неоднозначной в SDK iOS 18 | Скобки везде; `Darwin.cos/sin` — одна перегрузка, конверсии решаются детерминированно |
| 6 | `Missing bundle ID` при установке на симулятор | Рукописные Info.plist без `CFBundleIdentifier`; при `GENERATE_INFOPLIST_FILE: NO` Xcode его не подставляет | Стандартные ключи через переменные сборки в оба plist'а |
| 7 | Хост-приложение крашится на каждом тесте SwiftData-сьюты (48 с зависание → вотчдог) | Две причины: `makeContext()` выбрасывал `ModelContainer`, возвращая его `mainContext` (паттерн был и в исходных тестах — они никогда не запускались); якорный тест дёргал реальные AudioServices/UNUserNotificationCenter — они вешают headless-симуляторы | Контейнер удерживается свойством на время теста; системные side-effects тревоги пропускаются под XCTest (тесты проверяют журнал и латч, а не «бип») |
| 8 | **✅ ЗЕЛЁНЫЙ** — Build & Test и SwiftLint прошли на `277d6d0` (оба триггера) | Все 27+ юнит-тестов выполнились на симуляторе iPhone 16 | Пайплайн пройден целиком: генерация проекта → компиляция → линковка → установка → запуск → тесты → линт |

**Почему итерации через CI, а не локально:** в контейнере нет macOS/Xcode —
GitHub Actions выполнял роль компилятора. Поэтому каждая итерация
сопровождалась превентивными свипами (grep-скрипты по всем файлам на тот же
класс ошибки), чтобы не сжигать прогон на одну ошибку.

## 7. Ключевые архитектурные принципы, зафиксированные в коде

1. **Движки владеют фактами безопасности.** Запись MOB/якоря в журнал, хаптика,
   валидация — в `MOBEngine`/`AnchorWatchEngine`, вью только маршрутизируют.
   Новый экран не сможет «забыть» записать инцидент.
2. **`LogEvent.record` — единственный писатель журнала для движков.** Новое
   поле события добавляется в одном месте.
3. **Только опциональные добавления в модели** — до появления `VersionedSchema`
   (задача №1 следующей вехи).
4. **`.comingSoon()` — единственный способ показать недоделанное.** Ни одного
   включённого контрола без эффекта не осталось.
5. **Безопасность выше настроек:** активная вахта/MOB форсируют фоновую
   геолокацию независимо от тумблера (при выданном Always).

## 8. Что осталось (следующая веха, `hermes/04`)

1. `VersionedSchema` + `SchemaMigrationPlan`, замена `fatalError` на recovery —
   **до любого TestFlight**.
2. Реальное переключение единиц (снять Coming Soon с пикера).
3. UI добавления/редактирования обслуживания + напоминания.
4. PDF-экспорт.
5. Отложенные UI-консолидации: обобщить `PermissionCard` под карточки
   Weather/Today, общий `SelectableChip` для тегов и фильтров, скриншот-проверка
   бейджа «Скоро» на строках Form.
6. Известное ограничение: тревога при заблокированном экране требует Always
   (подсказка на экране вахты есть); дев-сборки, засиженные до гейта
   `--seed-demo`, хранят демо-данные до переустановки.
